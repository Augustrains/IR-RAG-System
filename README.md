# IR-RAG-System

一个面向信息检索教材与专业知识文档的 RAG 系统。项目围绕“教材类 PDF 难解析、图表信息难利用、检索与生成链路难闭环”这几个实际问题，构建了从文档清洗、结构化切分、索引构建、双路召回、精排重排序到前端问答展示的完整工程流程。

这个项目不是简单调用现成框架完成问答，而是针对教材场景做了较多定制化设计，包括双语检索、图表引用增强、跨页内容修复、可追溯回答输出，以及用于检索器和排序器训练的数据自动构建链路。

## 项目定位

本项目聚焦于信息检索（Information Retrieval, IR）教材场景，目标是让系统不仅能“答出来”，还要尽量做到：

- 检索命中更稳定：兼顾关键词匹配与语义匹配
- 回答更可追溯：输出页码、引用片段和关联图表
- 面向教材更友好：处理章节结构、枚举、图表标题、脚注等复杂版面信息
- 支持持续迭代：具备索引构建、训练数据生成、效果评测的完整闭环

## 项目亮点

### 1. 面向教材 PDF 的结构化解析，而不是直接粗暴切块

项目首先对 PDF 做定制化清洗与结构识别，包括：

- 页眉页脚、页码等噪声过滤
- 图表标题识别与跨行 caption 合并
- 同一视觉行碎片重新拼接
- 跨页内容修复，避免正文或图表说明被生硬截断

这一步的意义在于，后续检索和生成依赖的是“尽量接近原始语义结构”的文档块，而不是简单按固定长度截断的文本。

### 2. 自定义语义分块，融合结构先验与语义相似度

在切分阶段，项目没有只使用通用的固定窗口切分，而是额外设计了带结构先验的语义切分策略：

- 区分 section heading、sentence、list item、caption 等不同角色
- 结合句间语义相似度、词汇变化和结构规则决定是否切分
- 生成 parent/child 层级块，兼顾召回范围与上下文完整性

这让知识块更符合教材的表达逻辑，也为后续检索与重排提供了更好的输入单元。

### 3. 双语检索设计，解决中英文术语混杂场景

IR 教材中经常出现中英文术语混合表达，例如 `Boolean retrieval`、`postings list`、`precision/recall`。为此，项目在检索层做了双语设计：

- BM25 检索器按中英文分别建索引
- Milvus 检索器按中英文分别建立 hybrid collection
- 查询时不仅做原语言检索，还会翻译到另一种语言补查
- 最终对双路结果进行合并与去重

相比单语检索，这种设计更适合教材、论文、技术文档等术语跨语言混用的知识库场景。

### 4. 混合检索 + 精排，兼顾召回率与答案相关性

主链路采用了“多路召回 + 重排序”方案：

- 稀疏检索：BM25 负责关键词、术语和精确表达命中
- 稠密/混合检索：Milvus + BGE-M3 负责语义召回
- 结果融合：召回结果统一去重合并
- 精排：使用本地 BGE-M3 reranker 对候选文档进行相关性排序

这套方案的目的不是单纯堆模型，而是利用不同检索方式的互补性，在教材问答这种既需要术语精确匹配、又需要语义理解的场景中提高稳定性。

### 5. 图表增强检索，让文本块携带图表语义

这是项目里比较有特色的一部分。

很多教材问题虽然最终以文字提问，但答案往往依赖某个 Figure/Table 附近的说明文字。项目为此做了两层增强：

- 在解析阶段抽取图表 caption、图片路径、页码等元信息
- 在切块阶段识别正文中的 `Figure x.x` / `Table x.x` 引用，并把图表短描述注入到正文块中

这样做的好处是：

- 即使用户问题没有明确提到图名，相关块也更容易被召回
- 回答后处理时可以直接返回关联图表
- 系统从“只能回答纯文本”进一步走向“文本 + 图表线索”的多模态知识问答体验

### 6. 可追溯回答输出，而不只是返回一段文本

系统最终输出的不只是 answer，还会补充结构化信息：

- 引用页码
- 命中的原始文档片段
- 相关图表信息
- 本地图片路径

这意味着项目不仅关注“生成是否流畅”，也关注“回答是否能被验证”。对于面向教材、知识助手和企业知识库的 RAG 系统来说，这一点很重要。

### 7. 构建了训练数据与评测闭环，而不只是 Demo

除了在线问答主链路，项目还实现了用于持续优化的离线链路：

- 自动从教材分块中生成 QA 对
- 对高质量问题做 query generalization，增强训练集覆盖度
- 构建 reranker 训练数据
- 提供 retriever / reranker / 生成模型的独立评测脚本

这说明项目不是停留在“把系统跑起来”，而是已经往“可优化、可训练、可评估”的方向推进。

## 系统架构

```text
PDF教材
  -> 文档清洗与版面规则学习
  -> 跨页合并与图表/脚注信息保留
  -> 结构化语义分块
  -> MongoDB 存储结构化文档
  -> BM25 双语索引
  -> Milvus 双语 Hybrid 索引
  -> 用户问题输入
  -> BM25召回 + Milvus召回
  -> 候选去重合并
  -> BGE-M3 Reranker 精排
  -> 本地大模型生成回答
  -> 后处理输出答案、引用页码、关联图表、命中文档
```

## 核心流程

### 1. 文档构建链路

项目通过 `src/build_index.py` 将文档处理、索引更新和存储更新串成了一条完整流程：

- 解析 PDF 并清洗噪声内容
- 修复跨页断裂的正文和图表单元
- 进行结构化语义切分
- 将文档块同步到 MongoDB
- 更新 BM25 索引与 Milvus 向量库

### 2. 在线问答链路

主入口位于 `main.py`，当前实际流程为：

1. BM25 召回候选文档
2. Milvus hybrid 检索召回候选文档
3. 合并去重
4. 用 BGE-M3 reranker 精排
5. 将候选块拼接成带页码、图表、脚注信息的上下文
6. 调用本地 Qwen3-8B 生成回答
7. 对回答做后处理，提取引用编号、页码、图表与命中文档

### 3. 前端演示链路

项目包含一个可直接演示的 Flask 前端：

- 类 ChatGPT 风格会话界面
- 支持历史会话保存与恢复
- 支持展示引用文档片段
- 支持根据本地 `image_path` 展示关联图表

对于面试展示来说，这部分能比较直观地体现项目最终效果，而不仅是离线脚本。

## 创新点总结

如果从面试表达角度概括，这个项目的创新主要体现在以下几个方面：

- 不是通用 RAG 模板，而是针对教材场景做了 PDF 结构理解和图表增强。
- 不是单一路径检索，而是做了双语 BM25 + 双语 Milvus hybrid 的多路召回。
- 不是只返回答案，而是输出页码、文段和关联图表，强调可追溯性。
- 不是只做线上推理，而是补齐了 QA 生成、问题泛化、排序训练和效果评测闭环。
- 不是只强调模型调用，而是把数据处理、索引构建、检索优化、前端展示都串成了完整工程系统。

## 技术栈

- Python
- LangChain Document 抽象
- PyMuPDF / `fitz` 进行 PDF 解析
- MongoDB 存储结构化文档
- Milvus 构建混合检索索引
- BGE-M3 作为 embedding / hybrid retrieval 基座
- BGE-M3 Cross-Encoder 作为 reranker
- Qwen3-8B 作为本地生成模型
- Flask + HTML/CSS/JS 构建演示前端
